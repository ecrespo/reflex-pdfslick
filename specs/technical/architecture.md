## 1. Context

PDFSlick exposes its functionality primarily through the React **hook**
`usePDFSlick(url, options)`, which returns `{ viewerRef, thumbsRef, store,
usePDFSlickStore, PDFSlickViewer, PDFSlickThumbnails, error }`. The store is a
Zustand store; the `pdfSlick` instance inside it provides the imperative API
(navigation, zoom, rotation, print, download). PDF.js is browser-only and
manipulates `window`/`document`/canvas directly.

Reflex compiles Python components to a React/Next.js frontend with server-side
rendering by default. The central technical challenge is to wrap a
**hook-first, browser-only** library inside Reflex's **declarative, reactive,
SSR-capable** model so that:

1. Instantiation happens only on the client (no SSR access to `window`).
2. Props/options flow from Python state into `usePDFSlick`.
3. Viewer/store changes (page, scale, load, error) flow back into Reflex event
   handlers.
4. Imperative actions (go to page, zoom, rotate, print, download) are
   triggerable from Python.

## 2. Technical Goals

- **Correctness:** Python options serialize to exactly the JSON shape PDFSlick
  expects (camelCase keys; only serializable scalars cross the boundary).
- **SSR-safe:** no `window` access during server render; rendering deferred to a
  `next/dynamic` client-only import.
- **Reactivity:** store selectors drive Reflex events; prop changes re-render.
- **Maintainability:** a single thin JS wrapper; the Python layer (options,
  enums, models) is testable without a running frontend.

## 3. Why a local wrapper component (not a direct wrap)

A direct `rx.Component` wrapping `@pdfslick/react` is not viable because:

- The package's public surface is a **hook**, not a single drop-in component.
  Reflex wraps components, not hooks.
- `<PDFSlickThumbnails>` is a **render-prop / function-as-child** component; a
  Python lambda cannot be passed as a JS child render function.
- Several store fields are **non-serializable** (`Map`, `Date`, the `pdfSlick`
  class instance), so they cannot be synced directly into Python state.

Reflex's own guidance for hook-based / complex libraries is to **write a small
local React component** that provides the needed subset behind a simple
prop/event API, then wrap *that*. This is the approach adopted here.

## 4. Proposed Architecture

### 4.1 High-level diagram

```
┌─────────────────────────────────────────────┐
│  Python (Reflex app)                          │
│                                               │
│  pdf_slick(url=..., scale_value=...,          │
│            on_page_change=State.x, ...)       │
│  PdfSlickOptions / ScrollMode / SpreadMode    │
└───────────────┬───────────────────────────────┘
                │ compiles
                ▼
┌─────────────────────────────────────────────────────────┐
│  React / Next.js (generated by Reflex)                    │
│                                                           │
│  dynamic(import("PdfSlickWrapper"), { ssr: false })       │
│  <PdfSlickWrapper url={...} scaleValue={...}              │
│        onPageChange={...} onDocumentLoad={...} />          │
│        │ calls usePDFSlick(url, options)                  │
│        │ renders <PDFSlickViewer/> (+ optional thumbs)    │
│        │ usePDFSlickStore(selector) → useEffect → emit    │
└───────────────┬───────────────────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────────┐
│  @pdfslick/react → @pdfslick/core             │
│      → PDF.js (pdfjs-dist) + Zustand store    │
└─────────────────────────────────────────────┘
```

### 4.2 Components

| Layer | Artifact | Responsibility |
|---|---|---|
| Python API | `pdfslick.py` (`PdfSlick`, `pdf_slick`) | Declares props, events, imports; NoSSR. |
| Python types | `options.py`, `models.py` | `PdfSlickOptions` (PropsBase), enums. |
| JS wrapper | `js/PdfSlickWrapper.tsx` | Calls `usePDFSlick`, renders viewer/thumbs, subscribes to store, emits callbacks, imports CSS. |
| Demo | `pdfslick_demo/` | Reproduces the 7 examples. |

### 4.3 Reflex wiring (Phase 1)

```python
class PdfSlick(rx.NoSSRComponent):
    _wrapper = rx.asset("./js/PdfSlickWrapper.tsx", shared=True)
    library = f"$/public{_wrapper}"          # local source module
    tag = "PdfSlickWrapper"
    lib_dependencies = ["@pdfslick/react@4.0.0"]

    url: rx.Var[str]
    scale_value: rx.Var[str]
    single_page_viewer: rx.Var[bool]
    # ...
    on_page_change: rx.EventHandler[lambda page: [page]]
    on_document_load: rx.EventHandler[lambda num_pages: [num_pages]]
    on_scale_change: rx.EventHandler[lambda scale: [scale]]
    on_error: rx.EventHandler[lambda err: [err]]

    def add_imports(self):
        return {"": "@pdfslick/react/dist/pdf_viewer.css"}
```

- `rx.NoSSRComponent` emits the `next/dynamic(..., { ssr: false })` import so the
  wrapper only renders client-side.
- `library = f"$/public{rx.asset(...)}"` bundles the local `.tsx`; the npm
  package is declared in `lib_dependencies` so Reflex installs it.
- `add_imports({"": css})` imports the stylesheet once for its side effects.
- Snake_case props are auto-camelCased (`scale_value` → `scaleValue`).
- `rx.EventHandler[lambda x: [x]]` forwards the first JS callback arg to Python.

### 4.4 State bridge

The wrapper subscribes to store slices with `usePDFSlickStore((s) => s.field)`
and forwards changes through `useEffect`:

| Store field | Wrapper callback | Reflex event |
|---|---|---|
| `isDocumentLoaded` + `numPages` | `onDocumentLoad(numPages)` | `on_document_load` |
| `pageNumber` | `onPageChange(pageNumber)` | `on_page_change` |
| `scale` | `onScaleChange(scale)` | `on_scale_change` |
| `error` | `onError(error)` | `on_error` |

Only serializable scalars cross the boundary. Dates are converted to ISO
strings and Maps to plain objects/arrays inside the wrapper before emission
(see `../data-model/store-schema.md`).

### 4.5 Imperative controls (Phase 3)

PDFSlick actions live on the `pdfSlick` store instance and on its `.viewer`:

- Navigation: `pdfSlick.gotoPage(n)`.
- Zoom: setters `pdfSlick.currentScale = n` / `pdfSlick.currentScaleValue = "page-fit"`;
  `pdfSlick.viewer.increaseScale()` / `decreaseScale()`.
- Rotation: `pdfSlick.setRotation(deg)`.
- Modes: `pdfSlick.setScrollMode(ScrollMode.*)`, `pdfSlick.setSpreadMode(SpreadMode.*)`.
- Print: `pdfSlick.triggerPrinting()`.
- Download/Save: `pdfSlick.downloadOrSave()` / `download()` / `save()`.

Two options to drive these from Python (decided in Phase 3):

1. **Command prop** — a `command` Var (e.g. `{"action": "goto", "page": 3}`)
   the wrapper watches and applies, then clears. Simple, fully declarative.
2. **Imperative handle + `rx.call_script`** — expose the instance on a ref and
   call it via generated script. More direct but couples to generated names.

The command-prop approach is preferred for being declarative and Reflex-idiomatic.

## 5. SSR Strategy

PDF.js reads `window`/`document` at module load and runtime. The component is a
`rx.NoSSRComponent`, which makes Reflex emit:

```js
import dynamic from "next/dynamic";
const PdfSlickWrapper = dynamic(() => import("$/public/.../PdfSlickWrapper"), { ssr: false });
```

The wrapper therefore never executes during server render. The CSS import is a
side-effect import and is safe under SSR.

## 6. PDF.js worker

PDFSlick configures the PDF.js worker internally. Under Reflex's Next.js/webpack
bundler the worker (`pdf.worker.min.mjs`, pdf.js v6 is ESM/`.mjs`) must be
served with a version matching the bundled `pdfjs-dist` (6.0.227), or pdf.js
throws a version-mismatch error. Phase 1 verifies worker bundling in the demo
build; if needed, the wrapper sets
`pdfjsLib.GlobalWorkerOptions.workerSrc` before mounting.

## 7. Testing Strategy (summary)

- **Python unit tests** (no frontend): enums map to correct numeric values;
  `PdfSlickOptions` serializes snake_case → camelCase; package imports; the
  `PdfSlick` class declares the expected props/events.
- **Build/integration**: the demo app compiles and the viewer initializes
  (worker resolves; CSS applied).
- **Manual/e2e**: each example screen renders and reacts.

Detailed plan in `../plans/implementation-plan.md` and `../tasks/task-breakdown.md`.
